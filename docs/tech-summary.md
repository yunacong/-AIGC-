# 技术总结文档 · 电商 AIGC 内容经营助手

> 面向面试官的完整 AI 技术说明 · 含对比分析 · 版本 v1.0 · 2026-03-13

---

## 一、项目概述

**项目名称**：电商 AIGC 内容经营助手（E-commerce AIGC Content Operations Assistant）

**项目定位**：面向中小商家/达人的任务式工作流 Agent 产品，覆盖「商品理解 → 内容生成 → 图片优化 → 视频候选 → 内容诊断 → 复盘优化」完整电商内容经营闭环。

**技术栈**：React 18 + Vite + Tailwind CSS（前端）/ Dify（Agent 框架）/ Claude（LLM）/ 通义万象（图像）/ 可灵 AI（视频）

## 二、为什么选 Dify，不用 Coze / n8n / 其他

### 2.1 核心对比

| 对比维度 | Dify | Coze（字节）| n8n | FastAPI（纯代码）|
|----------|------|-------------|-----|-----------------|
| 开源程度 | 完全开源 ✅ | 闭源商业产品 | 开源 ✅ | N/A |
| 私有化部署 | Docker 一键部署 ✅ | 不支持 | 支持 ✅ | 支持 ✅ |
| 大模型支持 | 100+ 模型，含 Claude/GPT/通义 ✅ | 主要支持豆包，Claude 受限 | 通过 API 支持 | 任意 |
| 可视化工作流 | 完整拖拽编排 ✅ | 支持 | 支持 ✅ | ❌ 需要手写 |
| REST API 输出 | 原生支持，直接调用 ✅ | 支持 | 需要配置 Webhook | ❌ 需要自建 |
| Agent 能力 | 工作流 + 对话型 + ReAct ✅ | 工作流 + 插件 | 线性自动化为主 | 完全自定义 |
| 多模态支持 | Vision + 图像 API 工具节点 ✅ | 有限 | 弱 | 需要手写 |
| PM 可控性 | 可在后台直接调 Prompt ✅ | 可以 | 技术门槛较高 | 完全依赖工程师 |
| 部署成本 | 云端免费版可用，自部署廉价 ✅ | 免费有额度限制 | 自部署 | 需要服务器 |
| 社区活跃度 | GitHub 60k+ Star，活跃 ✅ | 商业产品 | 活跃 | N/A |

### 2.2 为什么不选 Coze

1. **模型支持受限**：Coze 以豆包模型为主，Claude 支持度较低。本项目核心需求是调用 Claude 进行高质量的中文内容创作，Coze 无法满足。
2. **闭源风险**：字节商业产品，API 政策可能随时调整，不适合需要稳定性的产品。
3. **国际化使用受限**：在某些场景下 Coze 访问不稳定。
4. **私有化能力缺失**：数据不能本地留存，对电商用户数据有合规隐患。

### 2.3 为什么不选 n8n

1. **Agent 能力偏弱**：n8n 本质是自动化工具，擅长简单线性流程，不适合需要条件分支、多 LLM 并行调用的复杂 Agent 场景。
2. **多模态支持差**：图像理解（Vision）和图像生成 API 接入复杂，没有开箱即用的工具节点。
3. **LLM 调用原生体验差**：需要自己封装 LLM API 调用，Prompt 管理困难，不如 Dify 有专门的 LLM 节点和 Prompt 调试界面。

### 2.4 为什么不选纯代码（FastAPI）

1. **MVP 阶段成本过高**：需要自己写所有工作流逻辑、Prompt 管理、API 路由，开发时间是 Dify 的 5-10 倍。
2. **迭代困难**：每次修改 Prompt 都需要代码变更和重新部署。
3. **缺乏可视化**：无法直观看到工作流节点和数据流向，调试困难。

### 2.5 Dify 选型结论

> Dify = **可视化编排** + **原生 LLM 支持** + **REST API 输出** + **私有化部署** + **PM 可操作**
> 这是 MVP 阶段实现「快速上线、灵活迭代、技术扩展」三者平衡的最优解。

## 三、涉及的所有 AI 技术

### 3.1 LLM 大语言模型

**选用模型**：Claude（Anthropic）

**具体版本**：
- **claude-sonnet-4-6**：主力模型，用于商品理解、内容生成、内容诊断、复盘优化
  - 优势：中文内容创作质量高，Prompt 指令遵循强，JSON 输出稳定，成本合理
  - 成本：约 $0.003/1K tokens，每次对话约 ¥0.02
- **claude-opus-4-6**：备选高质量版本，用于对内容质量要求极高的场景

**为什么选 Claude，不选 GPT-4o / 通义 / 文心**：

| 对比维度 | Claude Sonnet | GPT-4o | 通义千问 | 文心一言 |
|----------|---------------|--------|----------|----------|
| 中文内容创作 | ⭐⭐⭐⭐⭐ 极强 | ⭐⭐⭐⭐ 很强 | ⭐⭐⭐⭐ 很强 | ⭐⭐⭐ 良好 |
| 指令遵循（JSON输出） | ⭐⭐⭐⭐⭐ 极强 | ⭐⭐⭐⭐ 很强 | ⭐⭐⭐ 良好 | ⭐⭐⭐ 良好 |
| 电商文案风格 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 多模态（Vision） | ✅ 支持 | ✅ 支持 | ✅ 支持 | ✅ 支持 |
| 国内访问稳定性 | 需代理 | 需代理 | ✅ 直连 | ✅ 直连 |
| API 成本 | 中等 | 偏高 | 低 | 低 |
| 合规风险（国内） | 低（已备案） | 有 | 无 | 无 |

**实际选型逻辑**：优先选 Claude，因为内容创作场景对语言质量要求高，且 Dify 对 Claude 集成最成熟。如需国内直连访问，可切换为通义千问，Dify 支持一键切换模型。

**LLM 使用场景**：

| 功能模块 | LLM 任务类型 | Temperature | 说明 |
|----------|-------------|-------------|------|
| 商品理解 | 结构化分析提取 | 0.5 | 需要稳定性，低随机性 |
| 内容生成 | 创意写作生成 | 0.8 | 需要多样性，高随机性 |
| 图片诊断 | Vision 分析推理 | 0.3 | 需要客观准确，低随机性 |
| 内容诊断 | 分析评估 | 0.4 | 需要一致性判断 |
| 复盘优化 | 总结推理 | 0.6 | 需要综合分析能力 |

### 3.2 Prompt 工程（Prompt Engineering）

**核心设计原则**：

1. **角色设定（Role Prompting）**：每个 Agent 都赋予明确角色
   - 商品理解：「10年经验电商内容策划专家」
   - 内容生成：「生活化叙事达人 / 产品测评博主 / 电商运营」（3种角色）
   - 内容诊断：「抖音电商内容优化顾问」
   - 复盘优化：「电商内容经营教练」

2. **结构化输出（Structured Output）**：强制要求 JSON 输出
   ```
   严格按照以下 JSON 格式输出，不要有任何额外文字或 Markdown 代码块
   ```
   优势：前端可直接解析，不受模型「啰嗦」行为影响

3. **约束条件（Constraints）**：明确限制范围
   - 字数限制：「整体 150-200 字」
   - 内容规范：「不使用极限词，避免虚假宣传」
   - 格式规范：「附 3 个相关话题标签」

4. **多版本策略（Multi-version）**：内容生成用 3 套不同 Prompt 生成差异化版本
   - 场景共鸣型：生活化、情感化
   - 卖点功效型：数据化、对比化
   - 转化促销型：紧迫感、行动导向

**Prompt 评测方法**：

| 评测维度 | 评测方式 | 达标标准 |
|----------|----------|----------|
| JSON 解析成功率 | 100次调用统计 | ≥95% |
| 内容相关性 | 人工抽样评分 | ≥4/5分 |
| 字数控制 | 自动统计 | ±20%范围内 |
| 极限词检测 | 规则匹配 | 0违规 |
| 多版本差异度 | 相似度计算 | 各版本相似度<60% |

### 3.3 多模态 Vision（图像理解）

**技术原理**：将图片和文字同时输入多模态 LLM，模型能理解图片内容并用语言描述和分析。

**在本项目中的应用**：
- **图片诊断**：用户上传商品图 → Claude Vision 分析 → 输出「背景杂乱/主体偏小/光线偏暗」等诊断
- **诊断维度**：背景整洁度、主体突出度、光线质量、构图合理性、商品清晰度

**为什么用 Vision 而不是让用户自己描述**：
- 用户描述门槛高：中小商家不知道该说什么
- Vision 客观准确：减少主观偏差
- 体验更自然：上传图片比写描述更符合使用习惯

---

### 3.4 图像生成 API

**选用服务**：阿里云通义万象（Tongyi Wanxiang）

**为什么选通义万象**：
- 国内访问无需代理，延迟低
- 合规性好，适合电商商业场景
- API 接入简单，免费额度500张/月
- 商品图生成质量稳定

**与其他图像服务对比**：

| 服务 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| 通义万象 | 国内直连、合规、免费额度 | 创意性略低 | 本项目首选 |
| DALL-E 3 | 创意质量高 | 需代理、成本高 | 创意图生成 |
| Stable Diffusion | 开源免费、可定制 | 需要部署和优化 | 有技术团队时 |
| Midjourney | 艺术风格强 | 无 API、需要 Discord | 不适合程序调用 |

**图像生成 Prompt 设计**：
```
[商品名称] professional product photo, [风格要求] background,
commercial photography, high resolution, clean and minimal,
4K quality, professional lighting
```

### 3.5 视频生成 API

**选用服务**：可灵 AI（Kling，快手出品）

**为什么只做 5 秒视频（关键产品决策）**：

| 维度 | 5秒展示视频（本项目）| 完整长视频 |
|------|---------------------|-----------|
| 生成时间 | 30秒-3分钟 | 5-15分钟 |
| 生成成本 | 低 | 高 |
| 质量稳定性 | 较高 | 偏低 |
| 电商场景适配 | 完美（短视频素材、橱窗视频）| 一般 |
| MVP阶段可控性 | 强 | 弱 |

**技术关键点：异步任务机制**
```
1. 前端提交任务 → Dify 调用可灵 API → 返回 task_id
2. 前端每隔10秒轮询 → 查询任务状态
3. 视频就绪 → 返回视频 URL → 前端展示
```
这是实现视频生成功能必须理解的产品体验设计细节。

### 3.6 工作流 Agent（Task-based Agent）

**Agent 类型**：任务式工作流 Agent（Task-based Workflow Agent）

**区别于聊天式 Agent**：

| 对比维度 | 聊天式 Agent | 任务式 Agent（本项目）|
|----------|-------------|----------------------|
| 入口形态 | 自由对话框 | 结构化任务入口 |
| 用户要求 | 需要会写 Prompt | 选择任务即可 |
| 流程控制 | 用户主导 | 系统预设工作流 |
| 输出稳定性 | 低（依赖用户输入质量）| 高（固定模板） |
| 多模态组织 | 较难 | 清晰节点划分 |
| 适合用户 | 有 AI 使用经验 | 中小商家/达人 |

**6 个 Agent 的分工与关系**：
```
商品理解 Agent → 内容生成 Agent → 图片优化 Agent → 视频候选 Agent
                                                         ↓
                                            内容诊断 Agent
                                                         ↓
                                            复盘优化 Agent
                              （诊断结果可回流到内容生成形成闭环）
```

## 四、没有使用的技术及原因

### 4.1 RAG（检索增强生成）

**是什么**：将外部知识库（文档、案例库）和 LLM 结合，让模型检索相关知识后再回答。

**为什么没做**：
- MVP 阶段没有高质量知识库（爆款案例库、平台规则库）
- 没有知识库的 RAG 价值有限，效果不如精心设计的 Prompt
- 后续规划：积累内容数据后，引入爆款案例 RAG，实现「参考爆款生成内容」

**面试表达**：「RAG 不是不会，是主动判断当前阶段不具备前提条件，强上 RAG 反而会分散重点。」

### 4.2 Fine-tuning（模型微调）

**为什么没做**：
- 成本高：需要大量高质量标注数据（至少千级以上）
- 维护复杂：每次更新数据都要重新训练
- 当前阶段 Prompt 工程已能达到足够好的效果
- 后续规划：积累电商爆款内容数据后，可微调提升特定品类输出质量

### 4.3 向量数据库

**为什么没做**：当前无知识库，向量存储没有使用场景。后续若引入 RAG，配合 Dify 内置的向量存储功能即可。

## 六、面试高频问题 Q&A

### Q1：你这个项目用了哪些 AI 技术？

**A**：分三层：
- **语言层**：Claude LLM + Prompt 工程，用于商品理解、内容生成、诊断、复盘
- **视觉层**：Claude Vision（图片理解）+ 通义万象 API（图片生成）
- **视频层**：可灵 AI API（5秒展示视频生成）
- **编排层**：Dify 工作流 Agent，把以上能力按任务顺序串联

### Q2：为什么调用 Claude 而不是 GPT-4o？

**A**：主要有三点。第一，Claude 在中文内容创作场景下表现更稳定，电商文案的语气和风格更自然；第二，Claude 的指令遵循能力强，JSON 格式输出的成功率更高，减少了后处理的工作量；第三，Dify 对 Claude 的集成更成熟，包括流式输出和 Vision 能力都有完整支持。

### Q3：你的 Prompt 是怎么设计的？

**A**：我的 Prompt 设计主要围绕四个原则：
1. **角色设定**：给每个 Agent 明确的专家角色，例如内容生成里分别扮演「生活化达人」「测评博主」「电商运营」三种角色，同一个模型，不同角色，产出完全不同风格的内容；
2. **结构化输出**：强制要求 JSON 格式，方便前端直接解析；
3. **约束条件**：字数、合规要求、格式规范都写进 Prompt；
4. **多版本策略**：内容生成用 3 套独立 Prompt 并行生成 3 个差异化版本，给用户选择空间。

### Q4：如何评测 AI 输出质量？

**A**：评测分两层：
- **自动化评测**：JSON 解析成功率（≥95%）、字数控制准确率、极限词检测（0违规）
- **人工评测**：内容相关性（1-5分），抽样 30 条评估，达标分≥4分

迭代方法：持续 A/B 对比不同 Prompt 版本，收集用户选择哪个版本的行为数据，作为 Prompt 优化依据。

### Q5：为什么选 Dify 不选 Coze？

**A**：三个核心原因：第一，Coze 以豆包模型为主，Claude 支持有限，而本项目对内容质量要求高，必须用 Claude；第二，Dify 开源可私有化部署，适合有数据合规要求的电商场景；第三，Dify 的工作流编排能力更强，支持 Vision 工具节点和图像 API 节点，Coze 在多模态场景支持较弱。

### Q6：RAG 为什么不用？

**A**：RAG 的价值依赖于高质量知识库的建设。当前阶段还没有积累爆款案例库或平台规则库，强行上 RAG 效果不如精心设计的 Prompt 稳定。这不是能力问题，是阶段性的主动取舍。后续若积累足够的内容数据，会引入 RAG 实现「基于爆款案例的个性化推荐」。

### Q7：这个项目最难的技术点是什么？

**A**：有两个。第一是 **Prompt 工程的稳定性**，让模型在各种商品信息输入下都能输出符合格式的 JSON，这需要反复测试和迭代，在 Prompt 里加入格式约束、错误处理等容错机制；第二是 **多模态工作流的节点编排**，图片优化模块需要 Vision 节点诊断 + 并行图像生成节点 + 代码节点合并，节点间的数据传递和格式对齐需要仔细设计。

### Q8：这个项目如何扩展到更大规模？

**A**：分三阶段：
- **阶段 1（当前）**：Dify 云端 + Vercel，0 成本验证用户价值
- **阶段 2**：引入用户账号系统（Supabase），记录历史，接入 RAG（爆款案例库），实现「复盘」模块数据自动化
- **阶段 3**：将 Dify 迁移到自部署降成本，接入微信小程序，考虑用积累数据做 Fine-tuning 提升特定品类质量

## 七、一句话总结

**本项目的核心技术贡献不是接了多少 AI 模型，而是把「商品理解 → 内容生成 → 图片优化 → 视频候选 → 内容诊断 → 复盘优化」这条电商内容经营链路，通过任务式工作流 Agent 系统性地组织起来，让中小商家和达人无需了解 AI，选择任务就能完成内容经营全流程。**
